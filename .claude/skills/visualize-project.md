# Visualize Project

Generate architecture diagrams from the actual codebase. No guessing, no aspirational drawings -- read the code and diagram what exists.

## When to Use

- Onboarding to a new project
- After major refactors or new subsystems
- Before architectural discussions or reviews
- When someone asks "how does this project fit together?"

## Process

### Step 1: Scan

Read the project structure. If `/orient` or `/codebase-health` output exists from this session, reuse it. Otherwise scan:

- `package.json` (or equivalent manifest) for dependencies, workspaces, scripts
- Top-level directories and their purpose
- Config files (`.env*`, `docker-compose*`, `Dockerfile`, CI configs)
- Source file count (approximate -- use glob, not recursive find)
- Database setup (migrations, schema files, ORM config)
- Auth setup (middleware, auth config, session management)

### Step 2: Score Complexity

Add up signals from the table below. Each signal scores once, no double-counting.

| Signal | Score | How to detect |
|--------|-------|---------------|
| Database | +1 | `supabase/`, `prisma/`, `drizzle/`, `DATABASE_URL` in .env |
| Auth system | +1 | Auth middleware, next-auth, passport, auth config files |
| Single Docker | +1 | `Dockerfile` present, no multi-service compose |
| Multi-service Compose (3+ services) | +2 | `docker-compose.yml` with 3+ `services:` entries |
| Monorepo | +2 | `workspaces` in package.json, `packages/` or `apps/` dir |
| CI/CD | +1 | `.github/workflows/`, `.gitlab-ci.yml`, `.circleci/` |
| Multiple env files | +1 | 2+ `.env*` files |
| 100+ source files | +1 | Count of `.ts`, `.tsx`, `.js`, `.jsx`, `.py`, `.go`, `.rs` files |
| Migrations directory | +1 | `supabase/migrations/`, `prisma/migrations/`, `drizzle/migrations/` |
| Multiple AI provider keys | +1 | 2+ of `OPENAI_API_KEY`, `ANTHROPIC_API_KEY`, `MISTRAL_API_KEY`, etc. in env |

**Tiers:**

| Tier | Score | Diagrams generated |
|------|-------|--------------------|
| Simple | 0-1 | Module Overview |
| Standard | 2-4 | Module Overview, Data Flow, Route Tree (web apps only) |
| Complex | 5+ | All 6 diagram types |

### Step 3: Present Tier and Ask

Print the score breakdown and tier. Always offer three choices:

```
Complexity score: 4 (Standard)
  +1 Database (supabase/)
  +1 Auth (middleware.ts)
  +1 CI/CD (.github/workflows/)
  +1 Multiple env files (.env, .env.local)

Diagrams: Module Overview, Data Flow, Route Tree

Options:
  [U] Upgrade — generate Complex-tier diagrams instead
  [A] Accept — proceed with Standard
  [S] Skip — cancel, no diagrams generated
```

Wait for the user to choose before generating anything.

### Step 4: Generate Diagrams

Create `docs/architecture/diagrams/` if it does not exist.

Every diagram file uses this template:

```markdown
# [Diagram Title]

> **Snapshot**: Generated [YYYY-MM-DD] by `/visualize-project`. This is a point-in-time
> snapshot, not a living document. Re-run `/visualize-project` to regenerate from current
> codebase. For SVG: `npx -y @mermaid-js/mermaid-cli mmdc -i this-file.md -o this-file.svg`

## [Project Name] — [Diagram Type]

```mermaid
[diagram content]
```

## Notes

[Contextual notes about assumptions, exclusions, etc.]

<!-- Generated by /visualize-project • Re-run to update -->
```

Generate diagrams based on the tier:

**Module Overview** (`module-overview.md`) -- ALL tiers

- Mermaid `graph TD`
- 8-15 nodes max. Each node is a top-level module or directory with a clear responsibility.
- Edges show dependency direction (imports/calls).
- Group related nodes with subgraphs if it helps clarity.
- Do not diagram every file. Diagram the modules that matter.

**Data Flow** (`data-flow.md`) -- Standard+

- Mermaid `graph LR`
- Show how data moves: user request -> API -> service -> database -> response.
- Include external service integrations (AI providers, payment, email, etc.).
- Label edges with the transport (HTTP, WebSocket, SQL, SDK call).

**Route Tree** (`route-tree.md`) -- Standard+ (web apps only)

- Mermaid `graph TD`
- Map URL routes to their handlers/pages.
- Detect web app by: a web framework in dependencies (`next`, `express`, `fastapi`, `rails`, `django`, `hono`, `remix`, `nuxt`, `flask`, `koa`) OR a routing directory exists (`src/app/`, `src/pages/`, `routes/`, `app/`).
- Skip if none of these signals are detected (CLI tool, library, etc.) and note in the report: "Route Tree skipped -- no web framework or routing directory detected."
- Group by route prefix. Show middleware as intermediate nodes.

**Entity Relationships** (`entity-relationships.md`) -- Complex only, if database exists

- Mermaid `erDiagram`
- Diagram tables/collections and their relationships.
- Include cardinality. Skip if no database schema is detectable.

**Service Topology** (`service-topology.md`) -- Complex only, if multi-service

- Mermaid `graph TD`
- Show services, their ports, and how they communicate.
- Include databases, caches, queues, external APIs as nodes.
- Skip if the project is a single service.

**Feature Timeline** (`feature-timeline.md`) -- Complex only, if greenfield

- Mermaid `gantt`
- Estimated build phases based on the codebase structure and TODO/roadmap files.
- Generate ONLY if the project is greenfield: the working directory is empty, fewer than 10 source files exist, or the git history has 5 or fewer commits.
- Skip for established projects and note in the report: "Feature Timeline skipped -- project has [N] source files and [N] commits (not greenfield)."

### Step 5: Update CLAUDE.md

If a `CLAUDE.md` exists at the project root, add a reference line under the most appropriate section (usually Architecture or Quick Reference):

```
- **Architecture Diagrams**: `docs/architecture/diagrams/` — generated by `/visualize-project`
```

Do not add duplicates. If the line already exists, skip this step.

### Step 6: Report

Print a summary:

```
Generated N diagrams in docs/architecture/diagrams/:
  - module-overview.md
  - data-flow.md
  - route-tree.md

Tier: Standard (score 4)

These are point-in-time snapshots. Re-run /visualize-project to regenerate.
For SVG export: npx -y @mermaid-js/mermaid-cli mmdc -i <file>.md -o <file>.svg
```

## Cross-Platform Rules

- No bash-isms (`[[`, `pushd`, process substitution). Use POSIX-compatible commands.
- Use forward slashes in all file paths.
- Use `npx` (not global installs) for any CLI tools.
- Do not create or follow symlinks.
- Use Glob and Grep tools for file discovery, not `find` or shell globbing.

## Rules

- Diagram what exists, not what you think should exist.
- Never add nodes for planned-but-unbuilt features.
- Keep diagrams readable. Fewer nodes with clear labels beats comprehensive-but-unreadable.
- If a diagram type does not apply (no routes, no database), skip it and note why in the report.
- Re-read source files before diagramming. Do not diagram from memory or assumptions.
