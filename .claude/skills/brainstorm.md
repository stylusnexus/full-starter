# Brainstorm

Think before you build. This skill prevents the #1 failure mode: jumping straight to code without understanding the problem.

## Process

1. **Understand the request.** Read relevant code and docs first. Don't assume.
2. **Ask clarifying questions.** One at a time, not a wall of questions. Prefer multiple choice when possible.
3. **Propose 2-3 approaches.** Include trade-offs and your recommendation with reasoning.
4. **Get approval.** Present the design section by section. Don't proceed until the user agrees.
5. **Write a brief spec.** Save to `docs/specs/YYYY-MM-DD-<topic>.md` so the next session has context.
6. **Then build.** Now you can start coding.

## Rules

- Never skip straight to implementation on multi-step tasks
- One question per message (don't overwhelm)
- Always propose alternatives, even when one is obviously better
- The spec can be short (a few sentences for simple work), but it must exist
